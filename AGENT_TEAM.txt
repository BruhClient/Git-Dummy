EVO GIT - SUBAGENT TEAM
=========================

Five specialized Claude Code subagents were created under .claude/agents/ to
help with ongoing work on this PyQt5 git visualizer. Claude Code will
auto-delegate to the right one based on your request, or you can ask for one
by name (e.g. "use git-actions-debugger to look at ...").


1) git-actions-debugger
------------------------
File:  .claude/agents/git-actions-debugger.md
Model: sonnet
Tools: Read, Edit, Write, Grep, Glob, Bash, PowerShell, TodoWrite

Scope:
  Bugs in git operations and the in-app action workflow - anything where a
  git action (commit, push, pull, merge, stash, branch, revert, PR) behaves
  incorrectly, OR where the UI's handling of that action is broken (hangs,
  stuck-locked buttons, stale state, missing/misfiring dialogs).
  Covers: core/ops/*, core/git_tracker.py, ui/workers/commit_workers.py,
  and the QThread/action-orchestration layer in ui/commit_view.py.

Example triggers:
  - "The push button spins forever and never re-enables"
  - "After a merge conflict, the repo is left half-merged and no dialog appears"
  - "Switching branches sometimes shows the old branch's files"


2) canvas-visualizer-debugger
------------------------------
File:  .claude/agents/canvas-visualizer-debugger.md
Model: sonnet
Tools: Read, Edit, Write, Grep, Glob, Bash, PowerShell, TodoWrite

Scope:
  Bugs in the commit graph visualizer - lane assignment, branch attribution,
  edges, node/label rendering, orientation, and the minimap.
  Covers: ui/canvas/* (lane_algorithm.py, graphics_items.py, spatial_canvas.py,
  minimap.py, constants.py), GitTracker.graph_commits(), and commit_view.py's
  load_graph() pipeline.

  WARNING: top-level ui/spatial_canvas.py (~1,321 lines) is a stale, unused
  duplicate. The live canvas is ui/canvas/spatial_canvas.py.

Example triggers:
  - "Two branches are being drawn in the same lane and overlapping"
  - "Merged PR commits show up in the main lane instead of the feature lane"
  - "The minimap doesn't update when switching to left-right orientation"


3) ui-ux-polish
----------------
File:  .claude/agents/ui-ux-polish.md
Model: sonnet
Tools: Read, Edit, Write, Grep, Glob, Bash, PowerShell, TodoWrite

Scope:
  Visual and UX improvements - spacing, alignment, color/theme consistency,
  dialog layout and copy, loading/empty states, animations, overall polish.
  Covers: ui/panels/*, ui/dialogs/*, ui/components/*, styles/theme.py, and
  layout in main_window.py / repo_page.py / auth_page.py.

  Note: desktop PyQt5 app - verify changes by running
  "venv\Scripts\activate && python main.py" and looking at the screen,
  not a browser.

Example triggers:
  - "The settings panel buttons look misaligned and use a different blue"
  - "The clone dialog feels cramped and error states aren't styled"
  - "Can we add a nicer loading animation while the graph builds?"


4) code-quality-refactor
--------------------------
File:  .claude/agents/code-quality-refactor.md
Model: opus
Tools: Read, Edit, Write, Grep, Glob, Bash, PowerShell, TodoWrite

Scope:
  Structural cleanup - decomposing the ~2,900-line ui/commit_view.py,
  removing confirmed dead code, de-duplicating constants, enforcing
  core/ops/ and threading conventions, reducing duplication across core/
  and ui/. Must not change behavior - no test suite exists, so verify by
  running the app after each step.

Example triggers:
  - "commit_view.py is unmanageable, can we split it up?"
  - "Is ui/spatial_canvas.py at the top level actually used anywhere?"
  - "NODE_R and LANE_W are defined in multiple places with different values"


5) evo-git-polish
-------------------
File:  .claude/agents/evo-git-polish.md
Model: sonnet
Tools: Read, Edit, Write, Grep, Glob, Bash, PowerShell, TodoWrite

Scope:
  Generalist, whole-app sweeps - the opposite of the other four, which are
  each scoped to one layer and respond to a specific reported problem.
  Covers ui/, core/, styles/, auth/ - everything:
    - App-wide visual/UX consistency passes (same concerns as ui-ux-polish,
      but across multiple screens at once).
    - Proactive bug-hunting in areas nobody has reported issues in yet,
      across any layer (core/ops, threading/state flags, lane algorithm).
      Small self-contained bugs get fixed directly; deep findings get
      written up in action-catalog.md and routed to the right specialist.
    - Beginner-friendliness ideas: unexplained git jargon, raw git errors
      shown to users, defaults that assume git fluency. Small fixes
      (copy/tooltips) done directly; bigger ideas (onboarding flow, simple
      mode) get scoped and proposed rather than built unprompted.

Example triggers:
  - "Do a general pass over the app and tell me what feels rough or buggy"
  - "If someone who's never used git opened this app, what would confuse them?"
  - "Hunt for bugs across the whole app, not just the area I mentioned"


REFERENCE DOCS
==============
- .claude/agents/action-catalog.md
  Full catalog of all 22 user-triggerable git actions (UI button -> handler ->
  worker -> core/ops function -> exact git command sequence -> error handling),
  plus a ranked backlog of 6 known risk/inefficiency findings and a UI design
  recommendation for surfacing action details on toasts without compromising
  conflict-dialog info. git-actions-debugger treats this as its primary
  reference and known-issues checklist.


OVERLAPPING AREAS
=================
- git-actions-debugger vs canvas-visualizer-debugger: both touch
  commit_view.py's load_graph()/action code. The former owns ops/threading/
  state-flag correctness; the latter owns lane assignment/rendering
  correctness. A bug spanning both ("after pushing, the graph shows the
  wrong branch in the wrong lane") may need both.

- ui-ux-polish vs code-quality-refactor: both may edit the same UI files.
  ui-ux-polish changes are visual/UX only and must not change behavior.
  code-quality-refactor changes are structural and must preserve existing
  behavior/visuals exactly.

- evo-git-polish vs the other four: evo-git-polish is the generalist - app-
  wide sweeps rather than deep dives. It fixes small, self-contained findings
  directly; anything needing a deep dive in core/ops, threading, the lane
  algorithm, or a large commit_view.py restructuring gets written up in
  action-catalog.md and routed to the matching specialist.


DECISIONS MADE
==============
- No separate "Devil's Advocate" or "Lead Synthesis" agent was added - this
  conversation thread already synthesizes across the specialists, and the
  /code-review skill covers the critic role.
- No Agent/Task tool was given to any agent (no cross-agent delegation).
- No unrelated MCP tools (Canva, Gmail, Supabase, Scale Insights, Google
  Drive/Calendar) were inherited - each agent's tool list is explicit.
